# Admin Employee Modal Theory

## Bottom Line

Да, такую механику дать можно.

Но адаптация под ваш проект должна быть не "скопировать экран из картинки", а встроить его в текущую модель dashboard:

- админ открывает список сотрудников
- нажимает `Добавить сотрудника`
- получает модальное окно создания
- выбирает роль
- система показывает не хаотичный набор чекбоксов, а role-based preset доступа

Для вашего проекта это правильнее, чем сразу строить большой settings-экран с ручной матрицей прав на первом шаге.

## Почему модальное окно подходит именно вашему проекту

У вас уже есть операционный экран списка барберов, который логично станет экраном сотрудников.

Поэтому modal-flow хорошо ложится на текущий UX:

- не нужно уводить администратора на отдельную страницу
- создание сотрудника происходит в контексте уже открытого списка
- после сохранения админ сразу видит запись в таблице
- проще сохранить текущую архитектуру и не ломать навигацию раньше времени

То есть в вашем случае modal нужен как operational tool внутри раздела `Сотрудники`, а не как центр всей permission-системы.

## Как адаптировать референс под ваш проект

На скриншоте смешаны две разные задачи:

1. создание участника команды
2. управление матрицей ролей и прав

Для вашего проекта эти задачи лучше разделить.

### Что оставить из референса

- кнопку верхнего уровня `Добавить сотрудника`
- чистый admin-oriented интерфейс
- role-aware UX
- понятное разделение между данными сотрудника и его доступом

### Что не копировать один в один

- не помещать всю permission matrix внутрь модального окна создания
- не заставлять администратора руками отмечать десятки чекбоксов при каждом создании
- не делать settings-heavy сценарий первым экраном, если основной use-case это быстрый найм/добавление мастера или менеджера

Правильная адаптация:

- список сотрудников как основной экран
- modal для create/edit
- отдельный экран или вкладка `Роли и доступы` для общей матрицы permissions

## Рекомендуемая структура экрана

### Вариант, который лучше всего подходит сейчас

Раздел:

- `Сотрудники`

Верхняя панель:

- количество сотрудников
- фильтр по филиалу
- кнопка `Добавить сотрудника`
- кнопка `Обновить`

Таблица:

- имя
- телефон
- роль
- филиал
- специализация
- статус
- действия

Модальное окно:

- открывается по кнопке `Добавить сотрудника`
- используется и для `Создать`, и для `Редактировать`

### Что можно добавить позже

Отдельный admin-экран:

- `Настройки -> Роли и доступы`

Там уже можно держать:

- матрицу ролей
- список permissions
- пояснения, что может `manager`, `super-manager`, `barber`, `super-barber`, `admin`
- возможно, очень ограниченные custom overrides

Это ближе к скриншоту и логически должно быть вынесено отдельно.

## Каким должен быть create modal

Модалка не должна быть просто формой из пяти полей. Она должна создавать сотрудника как пользователя системы.

Рекомендуемые блоки:

### 1. Основные данные

- имя
- телефон
- логин
- временный пароль
- филиал

### 2. Роль

- `barber`
- `super-barber`
- `manager`
- `super-manager`
- `admin`

Здесь же нужно короткое пояснение под каждой ролью:

- кого она описывает
- какой уровень доступа даёт

### 3. Рабочий профиль

Показывается в зависимости от роли.

Для `barber` и `super-barber`:

- специализация
- рабочее место
- участие в очереди
- доступ к своей истории

Для `manager` и `super-manager`:

- филиал или филиалы
- доступ к статистике
- доступ к истории филиала
- управление сотрудниками

Для `admin`:

- глобальный доступ
- без привязки к одному филиалу, если это соответствует вашей бизнес-логике

### 4. Доступы

Здесь есть важный архитектурный момент.

В обычном create modal лучше не показывать полную таблицу чекбоксов.

Лучше показывать:

- выбранную роль
- краткое summary её прав
- список ключевых permissions только для просмотра

Например:

- `Сотрудники: чтение, создание, редактирование`
- `Статистика: филиал`
- `Очередь: только своя` или `Очередь: филиал`

Если позже нужен advanced mode, его лучше прятать в collapsible-блок:

- `Расширенная настройка доступа`

Но только для `admin` и, возможно, `super-manager`.

## Как связать это с вашей permission-моделью

Создание сотрудника через modal должно опираться не на ручное проставление доступа по месту, а на стабильную схему:

- выбирается `role`
- роль маппится в `permissions`
- permissions вычисляются сервером
- UI только показывает summary

Иначе админ каждый раз будет собирать доступы вручную, а проект быстро утонет в исключениях.

Для вашего кейса правильнее:

- роль как основной preset
- permissions как производный слой
- custom overrides только как редкое исключение

То есть модалка должна создавать не "случайного пользователя с набором галочек", а сотрудника с предсказуемой access-моделью.

## Какой UX будет самым удачным

### Для первого этапа

Самый практичный сценарий:

1. Админ открывает `Сотрудники`
2. Нажимает `Добавить сотрудника`
3. Выбирает роль
4. Видит адаптированные поля под эту роль
5. Видит summary будущего доступа
6. Сохраняет
7. Новый сотрудник появляется в таблице

Это быстро, понятно и соответствует текущему размеру проекта.

### Для второго этапа

Когда permission-layer стабилизируется, можно добавить рядом вторую сущность:

- вкладка `Сотрудники`
- вкладка `Роли и доступы`

Тогда:

- в `Сотрудники` живёт CRUD пользователей
- в `Роли и доступы` живёт матрица прав

Это уже ближе к референсу на скриншоте, но будет встроено в ваш продукт осмысленно, а не декоративно.

## Риски такого подхода

### 1. Перегруженная модалка

Если попытаться впихнуть в окно:

- аккаунт
- профиль
- филиалы
- график
- права
- ручные overrides

то modal станет слишком тяжёлым.

Решение:

- разделить на базовые поля и advanced settings
- часть вещей вынести в edit flow после создания

### 2. Смешивание ролей и permissions

Если в UI одновременно дать:

- выбор роли
- ручные чекбоксы по permissions

администратор быстро перестанет понимать, что реально управляет доступом.

Решение:

- на первом этапе только role presets
- overrides только потом и только ограниченно

### 3. Ошибки в branch-scope

Для `manager` и `super-manager` критично, каким филиалом они управляют.

Если это не оформить явно, можно случайно дать слишком широкий доступ.

Решение:

- branch scope должен быть отдельным обязательным блоком логики
- не прятать его внутри второстепенных полей

### 4. Неполная миграция терминологии

Если модалка будет называться `Добавить сотрудника`, а вокруг останутся:

- `Барберы`
- `/barbers`
- `barber profile`

появится путаница.

Решение:

- хотя бы на уровне UI переименовать раздел целиком консистентно

## Что я бы рекомендовал именно вам

Не делать копию скриншота один в один.

Для вашего проекта лучше такая последовательность:

1. Переименовать UI-домен `Барберы` в `Сотрудники`
2. Оставить основной экран списком сотрудников
3. У администратора сделать сильный create/edit modal
4. Внутри modal использовать выбор роли и summary permissions
5. Отдельно, позже, сделать экран `Роли и доступы` с матрицей, похожей на референс

## Главная мысль

Да, администратору можно и нужно дать модальное окно создания сотрудника.

Но в вашем проекте это должно быть:

- не центр permission-системы
- не копия HR SaaS интерфейса
- а быстрый admin-flow поверх нормальной модели `role -> permissions -> scopes`

Именно тогда решение будет масштабируемым, а не просто визуально похожим на референс.
